# 04 — Version Display and Build Stamp

## Why this step exists

When a user files a bug ("the popup is blank"), the first question is:
**which build?** Without a visible, copy-pasteable version + build id surfaced
in every UI surface (popup, options, in-page panel, logs, exported diagnostics
ZIP), the developer wastes hours guessing. This step pins the contract so the
same string appears everywhere and can never drift.

## Contract

1. **Single source of truth**: `manifest.json#version` is the canonical
   semver. Nothing else writes it by hand.
2. **Generated mirror**: a build script reads `manifest.json` and emits
   `src/shared/constants.ts` with:
   ```ts
   export const VERSION = "3.40.0";
   export const BUILD_ID = "3.40.0+ab12cd3";  // version + short git sha
   export const BUILD_TIME_ISO = "2026-06-05T03:14:00.000Z"; // UTC, set at build
   ```
3. **Mirror is committed** so the SW does not need to read `manifest.json` at
   runtime (SW has no filesystem).
4. **Pre-build verification** fails the build if `manifest.json#version`,
   `package.json#version`, and `constants.ts#VERSION` disagree.
5. **UI must surface** `BUILD_ID` on at least:
   - Popup footer (always visible).
   - Options page header.
   - In-page floating panel header (when present).
   - Diagnostics ZIP filename and `manifest.txt` inside the ZIP.
   - Every Code Red error log entry (`buildId` field).
6. **Click-to-copy**: clicking the `BUILD_ID` label copies the string to the
   clipboard so users can paste it into bug reports without typing.
7. **Reload triggers a stamp refresh** — see step `05-extension-reload-manual.md`;
   after `chrome.runtime.reload()`, the next render shows the new `BUILD_ID`
   without requiring a tab refresh of the popup.

## Reference generator

`scripts/stamp-version.mjs` (runs in `prebuild`):

```js
import fs from "node:fs";
import { execSync } from "node:child_process";

const manifest = JSON.parse(fs.readFileSync("manifest.json", "utf8"));
const pkg      = JSON.parse(fs.readFileSync("package.json", "utf8"));

if (manifest.version !== pkg.version) {
  console.error(
    `[stamp-version] CODE RED: path='manifest.json' missing='matching version' ` +
    `reason='manifest=${manifest.version} pkg=${pkg.version}'`,
  );
  process.exit(1);
}

let sha = "nogit";
try { sha = execSync("git rev-parse --short HEAD").toString().trim(); }
catch { /* no git in sandbox; keep 'nogit' */ }

const out = `// AUTO-GENERATED by scripts/stamp-version.mjs — do not edit.
export const VERSION = ${JSON.stringify(manifest.version)};
export const BUILD_ID = ${JSON.stringify(`${manifest.version}+${sha}`)};
export const BUILD_TIME_ISO = ${JSON.stringify(new Date().toISOString())};
`;
fs.writeFileSync("src/shared/constants.ts", out);
```

The header comment + an ESLint `no-edit` marker keep humans out of the file.
The script must be in `package.json#scripts.prebuild` so every `npm run build`
re-stamps automatically.

## Reference UI component (popup)

```tsx
import { BUILD_ID } from "@shared/constants";

export function VersionBadge() {
  const copy = () => {
    void navigator.clipboard.writeText(BUILD_ID);
  };
  return (
    <button
      type="button"
      onClick={copy}
      title="Click to copy build id"
      className="text-xs text-muted-foreground hover:text-foreground"
      data-build-id={BUILD_ID}
    >
      v{BUILD_ID}
    </button>
  );
}
```

Render it in the popup footer, options header, and panel header. The
`data-build-id` attribute lets E2E tests assert the visible value without
parsing the text node.

## Code Red log integration

Every error log row written via the namespace logger MUST carry `buildId`:

```ts
Logger.error("Injection.Failed", {
  path: "src/injected/sdk.ts",
  missing: "window.MyExt symbol",
  reason: "MAIN-world script returned before stamping global",
  buildId: BUILD_ID,
});
```

See step `11-error-logging-discipline.md` for the full Code Red shape.

## Diagnostics ZIP integration

The exported diagnostics bundle (see `mem://features/log-diagnostics-export`)
must:

- Name the file `diagnostics-<BUILD_ID>-<isoDate>.zip`.
- Include `manifest.txt` at the root with:
  ```
  BUILD_ID=3.40.0+ab12cd3
  BUILD_TIME=2026-06-05T03:14:00.000Z
  USER_AGENT=Mozilla/5.0 …
  EXPORTED_AT=2026-06-05T07:45:11.000Z
  ```

## Pitfalls

- **Reading `chrome.runtime.getManifest().version` everywhere** — works, but
  diverges from `BUILD_ID` (no git sha). Use `BUILD_ID` from `constants.ts`.
- **Hand-editing `constants.ts`** — the next prebuild overwrites your change.
  Bump `manifest.json` instead.
- **Showing `BUILD_ID` only on hover** — invisible to screenshot bug reports.
  Must be a visible text node.
- **Not stamping `BUILD_TIME_ISO` in UTC** — local timezone makes log
  correlation across users impossible. Always UTC ISO-8601.

## Acceptance

- [ ] `manifest.json`, `package.json`, and `src/shared/constants.ts` carry the
      identical version string after a fresh build.
- [ ] `BUILD_ID` is visible (not just hover) in popup, options, and in-page
      panel.
- [ ] Clicking the version badge writes the string to the clipboard.
- [ ] Every Code Red log row contains `buildId`.
- [ ] Diagnostics ZIP filename and `manifest.txt` both contain `BUILD_ID`.
- [ ] Pre-build script fails fast when versions disagree, with a Code Red log
      line that names the offending path.

  fixture manifest, asserts the emitted `constants.ts` matches the snapshot.

  asserts equality; runs in CI on every push.
- Popup component test — renders `<VersionBadge />`, asserts `data-build-id`
