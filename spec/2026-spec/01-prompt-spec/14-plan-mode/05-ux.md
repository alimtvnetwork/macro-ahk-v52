# 05 — Plan Mode UX

**Date:** 2026-06-02
**Task:** T90

## Entry points

1. **Toolbar button** "Plan" next to the prompts dropdown trigger.
2. **Keyboard shortcut** — host-configurable, suggested `Ctrl+Shift+P`.
3. **Slash command** in the ChatBox (if the host wires one): `/plan [count]`.

## Inline panel

A small popover anchored to the trigger:

```
┌─ Plan ────────────────────────┐
│ Steps  [ 10  ]   (1–50)       │
│ Prompt [ plan-default ▾ ]     │
│ ─────────────────────────────  │
│ [Cancel]              [Run ▸] │
└───────────────────────────────┘
```

- `Steps` is a numeric input bound to `PlanSettings.stepCount`; pre-filled, editable per-invocation without persisting unless the user opens Settings.
- `Prompt` dropdown lists prompts whose `tags` include `"plan"` plus the configured default; full list available via "Show all".
- `Run` enqueues exactly one task of `kind: "plan"`.

## During execution

- The queue widget surfaces the plan task with a distinct icon.
- On completion the host output area auto-scrolls and gains focus (`autoOpenResult`).
- On `Interrupted`/`hold`: same Resume affordance as Next.

## Empty state

If the configured `promptSlug` is missing, the panel shows a single error row with an inline "Choose another" button — running is disabled until resolved.

## Accessibility

- Panel is a focus trap; `Esc` closes; initial focus on the `Steps` field.
- Keyboard activation of `Run` is `Enter` while the panel is focused.
- The plan task's queue row announces start and completion via the same live-region as Next.
